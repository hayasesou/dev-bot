Attempt

Candidate

この再設計で特に重要な判断
1. WORKFLOW.md は正本、でも attempt 開始時に snapshot を取る

compact / 新 session の正式方針

あなたの案をそのまま contract にすると、こうなります。

原則

attempt id も維持する

branch / worktree も維持する

ただし workspace 状態が怪しいときだけ candidate branch から worktree を再生成する

これが一番安定します。
Codex App Server は compaction と steer と fork を持っていますが、dev-bot では “thread の記憶”より “artifact handoff” を信頼する べきです。

どのときに fresh session へ切るか

私は次を推します。

same candidate で steer 2 回以上

same candidate で repair loop 3 回以上

review repairable だが同じ失敗が続く

context が長くなり、次の目的が局所修正に変わった

minor repair reuse 条件に入る

どのときに same attempt のままにするか

lint / format / typecheck

medium 以下の narrow review fix

touched files が少ない

protected path に触っていない

scope drift なし

verification hard fail なし

どのときに new attempt にするか

review reject with high+

plan misalignment

unexpected files が多い

protected config violation

proof incomplete が構造的

hard checks fail が広範囲

この切り方が、あなたの「基本は fresh attempt、軽微修正だけ reuse」と一致します。

protected config 契約

この部分は frontmatter に明示した方がいいです。

方針

通常 issue では protected config は禁止。
専用 label が付いた issue だけ許可。
さらに issue 本文か artifact に allowlist を必須化。

これはシンプルで強いです。

おすすめの label 名

allow-protected-config

理由は、意味が一意で、他の label と混ざりにくいからです。

追加ルール

専用 label があっても、ルール緩和そのものは別禁止にした方がいいです。

つまり許可されるのは、

workflow に新しい check を追加

package.json に dependency を追加

CI matrix を増やす

のような変更であって、

lint rule を弱める

typecheck を無効化する

test command を甘くする

は label があっても原則禁止にするのが安全です。

この判定は protected_contract_ok と protected_path_violations に落とせます。

reject

accept

repairable

ただし minor repair policy に合う場合だけ

reject

必要なら planning committee を再実行する

committee only へ寄せる移行

これはあなたの方針で良いです。
committee only へ寄せるべきです。

winner 判定ロジックのおすすめ

あなたの「機械判定最優先」は全面的に賛成です。
私なら rank は 単純加点式より辞書順 tuple にします。

まず eligibility

hard checks pass

proof complete

protected path violation = 0

これを満たさない candidate は失格。

次に deterministic order

review severity vector

unexpected file count

missing required artifact count

verification failure count

diff size

duration_ms

この順で比較。

最後に tie-break

完全同点のときだけ Claude judge。

理由は、加点式より tuple の方が後から説明しやすいからです。
「なぜ alt1 が勝ったか」をそのまま日本語にしやすいです。

ここで確認したい質問

canonical 名称は run ではなく attempt に寄せてよいですか。私は run より attempt の方が Rework / replay と噛み合うと思っています。

plan_v2.json を 即 canonical にして、plan.json は compatibility view に落とす方針でよいですか。

review reject のうち、plan misalignment / scope drift を含む reject の場合だけ planning committee を自動再実行する方針にしたいのですが、これは入れますか。

私は、今回の contract を次の 6 本柱で固定するのが最もよいと思います。

canonical 単位は attempt

Rework は原則 new attempt

minor repair だけ same candidate reuse

winner selection は deterministic first

protected config は label + allowlist でのみ例外

この 6 本柱を WORKFLOW.md と artifact schema に入れると、candidate / rework / review reject / scope drift / protected config / session rollover が全部つながります。

1. まず用語を固定する

ここはかなり重要です。

Issue: GitHub Issue 単位の work item

Attempt: 1 回の end-to-end 実装試行

Candidate: 同一 attempt 内の並列候補

Session: underlying Codex / Claude の会話実体

Handoff: candidate を保ったまま session だけ切り替える継承

この定義にすると、あなたの方針がそのまま contract になります。

Rework = 基本 new attempt

primary / alt1 = 同一 attempt 内 candidate

compact / fresh session = candidate 継続、attempt 維持

crash recovery = same attempt, same candidate, same session lineage

つまり、session は runtime 実体、attempt は評価単位です。
この分離がないと、candidate 比較も replay も review reject の責務分解も全部曖昧になります。

5. 状態遷移の正式ルール
Ready

new attempt を作る

planning は approved 済み plan を読む

candidate_mode 判定で primary または primary + alt1

Rework

原則:

new attempt

candidate set は fresh

previous attempt は read-only evidence

例外:

minor_repair_reuse 条件を満たす場合のみ
same attempt / same candidate / new session

review = accept

winner 候補に残る

review = repairable

minor_repair_reuse 条件を満たす → same attempt / same candidate / new session

満たさない → new attempt / same plan

review = reject

plan_misalignment or scope_drift を含む
→ replan_reason.json を出す
→ committee planning 再実行
→ new plan + new attempt

それ以外
→ same plan + new attempt

candidate 完了

全 candidate の verification_result.json

全 candidate の review_result.json

全 candidate の proof_result.json

winner_selection.json
が揃って初めて winner を決める

push / PR

winner だけ

loser branch は local cleanup

compatibility views は winner だけ更新

hard_checks_pass != true

protected_path_violations > 0

unexpected_file_count

missing_required_artifact_count

verification_failure_count

10. candidate_policy.py の新ロジック草案

現状は rework_count または planner_confidence < 0.75 に加え、len(plan.design_branches) >= 2 を要求し、winner は単純加点です。

それを次へ寄せるべきです。

from __future__ import annotations
from dataclasses import dataclass

from app.contracts.artifact_models import (
    PlanV2,
    ProofResult,
    ReviewResult,
    ScopeAnalysis,
    VerificationResult,
)


@dataclass(frozen=True, slots=True)
class CandidateDecision:
    enabled: bool
    candidate_ids: list[str]


def should_enable_candidate_mode(plan: PlanV2, rework_count: int) -> bool:
    return rework_count >= 1 or plan.planner_confidence < 0.75


def decide_candidates(plan: PlanV2, rework_count: int) -> CandidateDecision:
    if should_enable_candidate_mode(plan, rework_count):
        return CandidateDecision(enabled=True, candidate_ids=["primary", "alt1"])
    return CandidateDecision(enabled=False, candidate_ids=["primary"])


@dataclass(frozen=True, slots=True)
class WinnerInput:
    candidate_id: str
    verification: VerificationResult
    review: ReviewResult
    scope: ScopeAnalysis
    proof: ProofResult
    diff_size: int
    duration_ms: int


def _severity_vector(review: ReviewResult) -> tuple[int, int, int, int]:
    counts = review.severity_counts
    return (
        counts.get("critical", 0),
        counts.get("high", 0),
        counts.get("medium", 0),
        counts.get("low", 0),
    )


def _eligible(candidate: WinnerInput) -> bool:
    if not candidate.verification.hard_checks_pass:
        return False
    if not candidate.proof.complete:
        return False
    if candidate.scope.protected_path_violations:
        return False
    return True


def _rank_tuple(candidate: WinnerInput) -> tuple:
    return (
        _severity_vector(candidate.review),
        candidate.scope.unexpected_file_count,
        len(candidate.proof.missing_artifacts),
        0 if candidate.verification.failure_type is None else 1,
        candidate.diff_size,
        candidate.duration_ms,
    )


def select_winner(candidates: list[WinnerInput]) -> str:
    eligible = [c for c in candidates if _eligible(c)]
    if not eligible:
        raise ValueError("No eligible candidate")

    ranked = sorted(eligible, key=_rank_tuple)
    return ranked[0].candidate_id

@dataclass(frozen=True, slots=True)
class WinnerInput:
    candidate_id: str
    verification: VerificationResult
    review: ReviewResult
    scope: ScopeAnalysis
    proof: ProofResult
    diff_size: int
    duration_ms: int


def severity_vector(review: ReviewResult) -> tuple[int, int, int, int]:
    counts = review.severity_counts
    return (
        counts.get("critical", 0),
        counts.get("high", 0),
        counts.get("medium", 0),
        counts.get("low", 0),
    )


def eligible(candidate: WinnerInput) -> bool:
    return (
        candidate.verification.hard_checks_pass
        and candidate.proof.complete
        and not candidate.scope.protected_path_violations
    )


def candidate_rank_tuple(candidate: WinnerInput) -> tuple:
    return (
        severity_vector(candidate.review),
        candidate.scope.unexpected_file_count,
        len(candidate.proof.missing_artifacts),
        0 if candidate.verification.failure_type is None else 1,
        candidate.diff_size,
        candidate.duration_ms,
    )


def select_winner(candidates: list[WinnerInput]) -> str:
    ranked = [c for c in candidates if eligible(c)]
    if not ranked:
        raise ValueError("No eligible candidates")

    ranked.sort(key=candidate_rank_tuple)
    return ranked[0].candidate_id

            "must_not_touch": plan_v2.get("must_not_touch", []),
            "protected_config_allowed": False,
        },
        "plan_context": {
            "goal": plan_v2["goal"],
            "verification_focus": plan_v2.get("verification_focus", []),
        },
        "completed_work": [],
        "remaining_work": [],
        "latest_failures": latest_failures,
        "latest_repair_feedback_refs": ["latest_repair_feedback.json"],
        "latest_review_delta_refs": ["latest_review_delta.json"],
        "input_artifacts": [
            "scope_contract.json",
            "verification_plan.json",
            "candidate_manifest.json",
        ],
    }
    self.state_store.write_candidate_artifact(
        issue_key, attempt_id, candidate_id, "session_handoff_bundle.json", bundle
    )
    return bundle
